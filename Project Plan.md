# DPS: Policy Document Processing Pipeline

## Overview

The DPS (Document Processing System) pipeline and notebook workflow transform policy DOCX files into chunk-friendly documents optimized for RAG retrieval.

The pipeline automates profiling (including word counts), control extraction, cross-reference extraction, heading fixes, splitting, and metadata injection. Notebook work handles transformation and semantic validation. Misc scripts handle supplemental audits.

## Pipeline Quick Reference

All automated steps run via `run_pipeline.py` from the project root. Config is `dps_config.xlsx` (primary) or `dps_config_fallback.yaml` (legacy fallback).

See user_guide.md > Running the Pipeline for CLI usage and standalone script commands.

**Pipeline steps:**

| Step | Script | What It Does | Output |
|------|--------|-------------|--------|
| 0 | `policy_profiler.py` | Scan all docs, extract metadata, classify types (A/B/C/D/E), score priority, count words | `output/0 - profiler/` |
| 1 | `acronym_finder.py` | Scan all docs for acronym candidates and generate audit Excel | `output/1 - acronyms/` |
| 2 | `extract_controls.py` | Pull structured control data with whitelist/blacklist filtering, multi-pattern ID matching, Published URL lookup, and CSV + Excel output | `output/2 - controls/` |
| 3 | `cross_reference_extractor.py` | Capture all cross-refs BEFORE any structural changes | `output/3 - cross_references/` |
| 4 | `heading_style_fixer.py` | Convert fake bold headings to real Word Heading styles | `output/4 - heading_fixes/` |
| 5 | `section_splitter.py` | Split fixed docs at H1 boundaries into RAG-sized sub-documents (greedy H2 fill to `max_characters`) | `output/5 - split_documents/` |
| 6 | `add_metadata.py` | Stamp sub-docs with identity metadata (name, URL, scope, intent, tags, custom tags) | `output/6 - metadata/` |
| 7 | `docx2md.py` | Convert .docx files to clean Markdown with YAML frontmatter | `output/7 - markdown/` |
| 8 | `docx2jsonl.py` | Convert .docx files to chunked JSONL for RAG/vector DB ingestion | `output/8 - jsonl/` |
| 9 | `validate_controls.py` | Validate all Step 2 controls are present in Step 5 split documents; produces review workbook for human annotation | `output/9 - validation/` |

**Always run Step 0 first.** Step 4 reads from Step 3's output. Step 5 reads from Step 4's output. Step 6 reads from Step 5's output. Steps 7 and 8 read from Step 6's output (or Step 4's, configurable). Steps 1, 2, 3 read from `input/` directly.

## Input File Classification

Pipeline input files fall into two categories: **human-validated** files that a person has reviewed and confirmed, and **pipeline-generated** files that are unvalidated machine output. The distinction matters because validated files are authoritative and should not be deleted or regenerated without re-review, while pipeline output can always be regenerated from source.

### Human-Validated Input Files

See [`Misc/Manual_Validation.md`](Misc/Manual_Validation.md) for the full table of human-validated input files and the validation flow.

### Unvalidated Pipeline Output

Everything in `output/` is machine-generated and can be regenerated by re-running the relevant step. **Safe to delete** — the pipeline will recreate them. Useful for review but not authoritative until a human validates them.

| Output | Generated By | Purpose | Feeds Into |
|--------|-------------|---------|------------|
| `output/0 - profiler/*` | Step 0 | Document metrics, classification, priority scores | Step 6 (profiler data), human review |
| `output/1 - acronyms/acronym_audit.xlsx` | Step 1 | Raw acronym candidates — **unreviewed, may contain false positives** | Human review → `input/Acronym_Definitions.xlsx`; fallback for Step 6 tags if no definitions file |

## Tag System Architecture (Step 6)

Step 6 generates tags from **five sources**, merged in this order (duplicates auto-removed, first occurrence wins):

| # | Source | Example Output | Config Key | Per-Document? |
|---|--------|---------------|------------|---------------|
| 1 | **Doc type** from profiler (Step 0) | `Type-C` | `metadata.tags.include_doc_type` | Yes (auto) |
| 2 | **Sections found** from profiler | `has-scope`, `has-controls` | `metadata.tags.include_sections_found` | Yes (auto) |
| 3 | **Unique acronyms** from Acronym Definitions | `AC`, `MFA`, `IR` | `metadata.tags.max_acronym_tags`, `max_tag_doc_count` | Yes (auto) |
| 4 | **Custom tags** from "Custom Tags" sheet | `CUI`, `FedRAMP-High`, `Priority` | *(automatic — sheet in Acronym Definitions file)* | Yes (user-defined) |
| 5 | **Static tags** from config | `InfoSec`, `GCC-High` | `metadata.tags.static_tags` | No (all docs) |

**Result example:** `Type-C, has-scope, has-controls, AC, MFA, CUI, FedRAMP-High, InfoSec`

### Consolidated Input File Design

All per-document metadata lives in **one workbook** — `input/Acronym_Definitions.xlsx`:

| Sheet | Columns | Purpose | Used By |
|-------|---------|---------|---------|
| **Acronym Definitions** | Document \| Acronym \| Definition \| Status \| Notes | Human-verified acronym glossary. Status column lets reviewers mark rows without deleting them. | Steps 6, 7, 8 |
| **Per Document** | Document \| Acronym \| Occurrences \| Found In \| Definition(s) | Per-doc occurrence detail | Steps 6, 7, 8 (fallback) |
| **Custom Tags** | Document_Name \| Tags | Per-document custom tags (comma-separated) | Steps 6, 8 |

Step 1's audit output (`acronym_audit.xlsx`) includes all these sheets — including a pre-populated "Custom Tags" sheet with document names and format instructions filled in. Users curate the audit output into `Acronym_Definitions.xlsx` and fill in custom tags at the same time.

The **only other input Excel** is `input/Doc_URL.xlsx` (document name → SharePoint URL mapping).

### Custom Tags (Per-Document)

Users can assign arbitrary tags to specific documents without affecting auto-generated tags, acronym discovery (Step 1), or control validation (Step 9). No config changes or extra files needed — just add a "Custom Tags" sheet to the Acronym Definitions file.

**Setup:**
1. Open `input/Acronym_Definitions.xlsx` (or copy from Step 1's `acronym_audit.xlsx`)
2. Go to the "Custom Tags" sheet (or create one)
3. Fill in: `Document_Name` | `Tags` (comma-separated per row, e.g. `access control, authentication, CUI`)

### Acronym & Tagging Process: End-to-End Failure Point Map

This section documents every point where the acronym discovery → human curation → tag generation → metadata injection chain can silently fail. The chain crosses four scripts and two human-edited Excel files, making it the most fragile data path in the pipeline.

#### Data Flow

```
Step 1 (acronym_finder.py)
    │
    ▼
acronym_audit.xlsx          ◄── 7 sheets, Status column, instruction rows
    │
    │  ┌──────────────────────────────────────────────────┐
    │  │  HUMAN REVIEW (highest failure risk)             │
    │  │  1. Set Status = Confirmed / False Positive      │
    │  │  2. Fill in missing definitions (yellow rows)    │
    │  │  3. Fill in Custom Tags (comma-separated)        │
    │  │  4. Save as input/Acronym_Definitions.xlsx       │
    │  └──────────────────────────────────────────────────┘
    │
    ▼
input/Acronym_Definitions.xlsx
    │
    ├──→ Step 6 (add_metadata.py)     → tags in sub-document metadata blocks
    │       Reads: Acronym Definitions + Custom Tags sheets
    │       Filters: Skips Status="False Positive" rows
    │
    ├──→ Step 7 (docx2md.py)          → acronym lists in YAML frontmatter
    │       Reads: via excel_lookup_list/dict config sources
    │       Matches: match_doc_name() substring matching
    │
    └──→ Step 8 (docx2jsonl.py)       → Acronyms + tags fields in JSONL chunks
            Reads: Acronym Definitions + Custom Tags sheets
            Filters: Skips Status="False Positive", skips rows without definitions
            Warns: Lists unmatched documents at end of run
```

#### Failure Points by Category

##### 1. Document Name Matching (HIGHEST RISK — affects all downstream steps)

Every consumer script must match document names from the Excel against actual filenames. The matching uses `normalize_doc_name()` (strips extension, lowercases, collapses `_/-/space` into spaces) and `match_doc_name()` (substring containment in both directions). This is **deliberately fuzzy** to tolerate naming variations, but creates risks:

| Failure Mode | What Happens | How to Detect | How to Prevent |
|---|---|---|---|
| **Typo in Document column** | Acronyms/tags silently not applied to that document | Step 8 now prints "N document(s) had no matching acronym definitions" at end of run | Use Step 1's pre-populated names — don't retype |
| **Short name matches wrong doc** | e.g. "IR" in Acronym_Definitions matches "Incident_Response" AND "IR_Plan" | Inspect metadata output for unexpected acronym assignments | Use full document names, not abbreviations |
| **Filename changed after Step 1** | Document column has old name, no match | Step 8 unmatched-doc warnings | Re-run Step 1 after renaming input files |
| **Extension included/excluded inconsistently** | `normalize_doc_name()` strips `.docx`, but if someone types `.doc` it won't strip | Manual review of Document column | Always use names without extensions, or always with `.docx` |

##### 2. Human Curation Errors (Acronym Definitions Sheet)

| Failure Mode | What Happens | How to Detect | How to Prevent |
|---|---|---|---|
| **Definition left blank** | Step 8 (JSONL) excludes that acronym — it won't appear in chunk metadata | Step 8 now logs "N without definitions skipped" | Fill in definitions for all confirmed acronyms, or accept the exclusion |
| **Row deleted instead of marking False Positive** | Acronym reappears on next Step 1 re-run; no audit trail | Compare row counts between runs | Use the Status dropdown instead of deleting rows |
| **Wrong definition entered** | Incorrect expansion propagated to all RAG chunks for that document | Spot-check JSONL output for high-frequency acronyms | Cross-reference with Global Summary sheet |
| **Status column ignored (left blank)** | All rows treated as confirmed (blank = keep) | Review Status column before saving to input/ | Use the dropdown — Confirmed, False Positive, or Needs Review |

##### 3. Custom Tags Sheet Errors

| Failure Mode | What Happens | How to Detect | How to Prevent |
|---|---|---|---|
| **Tags not comma-separated** | Entire cell treated as one tag (e.g. "access control authentication" becomes a single tag) | Check tag arrays in JSONL output | Follow the instruction row format: `tag1, tag2, tag3` |
| **Leading/trailing spaces in tags** | Tags may appear as ` CUI` instead of `CUI` | Both consumers `.strip()` each tag, so this is handled | N/A — auto-cleaned |
| **Document name doesn't match** | Tags silently not applied | Step 8 unmatched-doc warnings | Don't edit the pre-populated Document_Name column |

##### 4. Script-Level Failures

| Failure Mode | Script | What Happens | Resolution |
|---|---|---|---|
| **Acronym file path wrong in config** | Steps 6, 8 | Empty acronym/tag data, graceful fallback | Check `metadata.tags.acronym_definitions_file` and `docx2jsonl.acronym_definitions_file` paths |
| **Sheet renamed in Excel** | Steps 6, 8 | Sheet not found, data skipped | Keep sheet names as "Acronym Definitions" and "Custom Tags" (case-insensitive match) |
| **Column header renamed** | Steps 6, 8 | Column not matched, empty data | Keep headers as Document, Acronym, Definition, Status, Notes (flexible substring match, but avoid ambiguous headers) |
| **JSONL field name mismatch** | Step 8 | Field name hardcoded in docx2jsonl.py, not from config | Rename in the script source code if needed (see CLAUDE.md > JSONL Field Name Caveat) |
| **Tag file vs. acronym file confusion** | Step 8 | `docx2jsonl.tag_file` overrides the Custom Tags sheet fallback | Leave `tag_file` empty to use the Custom Tags sheet in Acronym_Definitions.xlsx (default) |

##### 5. Instruction Row Handling

Step 1 now generates instruction/helper rows (row 2) in the Acronym Definitions and Custom Tags sheets. These rows start with `(` and use italic gray formatting. Downstream consumers (Steps 6 and 8) skip rows where the Document column starts with `(`. If a user adds their own parenthetical notes in the Document column, those rows will also be skipped.

## Phase 1: Document Profiling

**Run:** `python run_pipeline.py --step 0`

Step 0 (`policy_profiler.py`) produces per-document:

- Total paragraph count and approximate page count (paragraph count / 30)
- All paragraphs using Word built-in Heading styles (Heading 1/2/3) vs bold-text-no-style paragraphs
- Custom named styles functioning as structural headings
- Table count, row/column counts per table, first row content per table
- Total character count and character count per H1 section
- Table density ratio (characters in tables vs total characters)
- Merged cells, nested tables, text boxes, tracked changes, comments, images, embedded objects
- Password protection, IRM protection
- Five standard section detection (Purpose, Scope, Intent, Controls, Appendix) via fuzzy match on H1 text
- Cross-reference patterns: "See Section", "refer to", "as described in", hyperlinked cross-references

### Section Terminology

Two distinct concepts are tracked for every heading-delimited section:

| Field | What it is | Example |
|-------|-----------|---------|
| **`heading_text`** | The raw text as it appears in the `.docx` file — exactly what the author typed | `"1. Purpose and Applicability"`, `"Scope of This Policy"` |
| **`standard_section`** | The normalized semantic label the profiler assigns via fuzzy match | `purpose`, `scope`, `intent`, `controls`, `appendix`, or `none` |

`heading_text` is *what the document calls it*. `standard_section` is *what we decide it means*. Headings that don't match any of the five standard labels get `standard_section = "none"` and are listed separately as `unique_sections` in the inventory.

The match is substring-based and case-insensitive: if any configured lookup term appears in (or contains) the heading text, that label is assigned.

See `user_guide.md > Section 3: Document Structure Detection` for customizing lookup terms.
- Priority score — combines table count, cross-ref count, fake headings, page count, missing sections, merged cells, oversized flag

Output format: `document_inventory.xlsx` (master spreadsheet) + `document_profiles.json` + CSVs per data type.

See `user_guide.md > Configuration Reference` (Sections 3, 4, 8, 9) for tuning profiler settings.

## Phase 2: Document Type Classification

Classification is automatic in Step 0. Each document gets a type in the inventory output.

### Classification Rules

- **Type C (Hybrid) — DEFAULT:** 10-40% table content. Mix of prose and tables. Most docs land here. Base notebook instructions are written for this type.
- **Type A (Table-Heavy Control Docs):** >40% content in tables. Control matrices, applicability tables, configuration baselines.
- **Type B (Prose-Heavy Intent Docs):** <10% table content. Long narrative sections, rationale, risk context.
- **Type D (Appendix-Dominant Docs):** >60% appendix material regardless of table density in the policy body.
- **Type E (Unclassified):** Does not fit A-D. Stop. Review manually before processing. Never force-fit.

Thresholds are configured in `dps_config.xlsx` under `classification`. Documents that fall between thresholds default to Type C.

Review the Step 0 inventory for type assignments before proceeding. Override any misclassifications manually — the type determines which notebook addendum to use in Phase 3. See [`Misc/Manual_Validation.md`](Misc/Manual_Validation.md) for classification descriptions.

## Phase 4: Validation

Validation runs two tracks: automated structural checks (Step 0 word counts + additional scripts) and semantic checks (Notebook 3 + human review).

### Automated Structural Validation

**Step 0** (`python run_pipeline.py --step 0`) now includes word counts per document in the profiler output as a baseline QA check. For ad-hoc word counting on split sub-documents, use `python scripts/word_counter.py`.

For full structural validation, generate a batch validation script that checks per restructured document:

- Missing heading styles (fake bold headings remaining)
- Remaining tables (count per doc)
- Document length (character count vs 36,000 limit)
- Summary paragraph existence (first paragraph keyword check)
- Cross-reference pattern detection ("See Section", "refer to" remaining)
- File naming convention compliance
- Self-identifying context sentence presence per section

Output: Pass/fail scorecard per document as CSV.

### Semantic Validation (Notebook 3)

Notebook 3 runs these checks that require model judgment:

1. **Self-identification:** Does every section begin with policy name + topic sentence?
2. **Header quality:** Are all headers descriptive and policy-named?
3. **Cross-reference completeness:** Any remaining unreplaced references?
4. **Table elimination:** Any remaining tables?
5. **Intent-to-control linkage:** Sample 5 control paragraphs for dual presence of requirement + intent
6. **Isolation test:** Pick 3 random paragraphs. Read each with zero context. Can you identify: which policy, what topic, what requirement?
7. **Document length:** Under 36,000 characters?
8. **Summary paragraph completeness:** Policy area, control count, frameworks, applicability, review date all present?

#### How `confirmed_controls.csv` Is Built

1. Load all rows from `controls_output.csv` (the original Step 2 extraction)
2. Load all rows from the "Validation Review" sheet in the annotated workbook
3. Index reviewed rows by `(control_id, source_file)` → `Validation Status`
4. **Keep:** controls marked "Correct" or left unreviewed (assumed correct)
5. **Exclude:** controls marked "Wrong-FalsePositive"
6. **Append:** controls from the "Add Missing Controls" sheet (with `extraction_source` set to `Manual-Review`)
7. All other wrong statuses (Wrong-Description, Wrong-Guidance, etc.) are kept in the CSV but flagged in the report

#### Pattern Detection & Config Suggestions

The script analyzes reviewer feedback for systematic error patterns and generates actionable config suggestions:

| Pattern Detected | Threshold | Suggested Action |
|-----------------|:---------:|------------------|
| Multiple false positives sharing a control ID prefix | 3+ from same prefix | Add `PREFIX-*` to `control_extraction.blacklist` in `dps_config.xlsx` |
| Multiple false positives from one section header | 3+ from same section | Add section name to `SUSPECT_SECTIONS` in `validate_controls.py` |
| Missing controls with IDs not matching current regex | Any unmatched | Add new `control_id_pattern` to `dps_config.xlsx` |
| `GUIDANCE_IN_DESC` flag correlating with description errors | 3+ correlated | Review `guidance_keywords` in `dps_config.xlsx` |

#### Improvement & Extension Points

- **Adding new Validation Status values:** Add the new status string to `validate_controls.py`'s dropdown list, then update `build_confirmed_controls()` in this script to handle the new status (currently, anything not `Wrong-FalsePositive` is kept). Update `analyze_error_patterns()` if the new status needs pattern analysis.
- **Adding new suggestion types:** Add detection logic in `analyze_error_patterns()` (append to `analysis["suggestions"]` with `type`, `reason`, `action`, optional `value`), then add a formatting block in `write_suggested_config()` for the new type.
- **Adding new flag correlation analysis:** In the `Flag accuracy` section of `analyze_error_patterns()`, the script already correlates all flags with all error statuses generically. To add targeted analysis for a specific flag, follow the `GUIDANCE_IN_DESC` pattern — check for threshold and append a suggestion.
- **Integrating `confirmed_controls.csv` downstream:** The CSV uses the same schema as `controls_output.csv`. Any step that reads controls (e.g., `validate_controls.py`) can be pointed at `confirmed_controls.csv` instead by changing the input path in config. This creates a progressive refinement cycle.
- **Batch re-validation:** After applying config suggestions and re-running Step 2, run Step 9 again and diff the new `validation_review.xlsx` against the previous `feedback_report.txt` to measure improvement.

## Known Risks & Missing Features

### Where Policy Intent Can Drift

| Pipeline Stage | Risk | How Intent Drifts |
|---|---|---|
| **Heading Style Fixer** | Medium | A fake heading misclassified as H1 creates a false split boundary — orphaning context that belonged with the previous section. A real H1 missed means two distinct topics merge into one chunk, diluting both. |
| **Section Splitter (H1 → sub-docs)** | High | Splitting at H1 severs the narrative arc. A "Purpose" section may state intent that qualifies every control beneath it. Once split, downstream chunks lose that qualifying language. The preamble helps, but preamble ≠ full Purpose section. |
| **Section Splitter (H2 fallback)** | Medium | When an H1 section exceeds `max_characters`, H2 sub-sections are accumulated greedily until the limit is crossed, then split at the next H2 boundary. Tightly coupled subsections that together exceed the limit (e.g., a large "Scope" + "Applicability") will still land in separate docs — but the greedy approach keeps them together whenever they fit. Tune `max_characters` in `dps_config.xlsx` if coupled sections are being separated. |
| **Preamble Prepend** | Low-Medium | The preamble is content before the first H1. If a doc puts its actual intent/scope *after* H1 (common in appendix-dominant Type D docs), every sub-doc gets a hollow preamble — title and version but no policy context. |
| **Table Flattening (Notebook 2)** | High | Control matrices encode relationships via row/column position. Flattening to prose requires interpreting those relationships. A "Yes" cell under "Encryption Required" next to "PII at Rest" becomes a sentence — and the sentence's wording *is* the policy now. |
| **Cross-Reference Inlining (Notebook 2)** | Medium | Replacing "See Section 4.3" with the actual text of 4.3 can change meaning if 4.3 is paraphrased or truncated. Over-inlining bloats chunks; under-inlining leaves dangling pointers. |

### Missing from the Pipeline

1. **Sub-Document Titling Strategy:** Output files named `[OriginalName] - [Heading1Text].docx` produce generic names like `*- Controls.docx` across many source docs. Needs a convention encoding source policy name + specific topic.
2. **Duplicate/Overlap Detection:** No mechanism detects when two sub-docs from different source policies cover the same topic. Copilot retrieves both and may give contradictory answers.
3. **Chunk Identity Metadata:** ~~Beyond preamble, nothing marks *where in the original doc* a chunk came from.~~ **Addressed by Step 6 (`add_metadata.py`).** Each sub-doc now gets a metadata block with document name, URL, scope, intent, and tags. URLs are sourced from `input/Doc_URL.xlsx` — one file shared by both Step 2 (Published URL column in controls export) and Step 6 (metadata injection). Steps 7 and 8 propagate this metadata to Markdown frontmatter and JSONL chunks respectively. Effective date and parent heading chain are not yet included — add as custom fields in `metadata.fields` config if needed.
4. **Version/Staleness Tracking:** When a source doc is revised, no mechanism identifies which sub-docs need regeneration. `split_manifest.csv` needs a hash or timestamp per sub-doc.
5. **Round-Trip Validation:** No check confirms the union of all sub-doc content equals the original (minus formatting). Missing or duplicated paragraphs from XML deep-copy edge cases go unnoticed.
6. **Table Handling at Split Boundaries:** Tables attributed to the section where they start. A table spanning an H1 boundary goes entirely into the first section's sub-doc even if it semantically belongs to the next.
7. **Max Doc Count Guardrail:** No check enforces the 400-doc SharePoint/Copilot source limit. The splitter should track cumulative sub-doc count and warn before exceeding the ceiling.

Options to stay under 400: merge small adjacent sections before splitting, consolidate Type D appendix docs into shared reference docs, or exclude boilerplate sections (Document History, Approval Signatures) from output.

## Excel Config System (`dps_config.xlsx`)

The pipeline is configured through a single Excel workbook (`dps_config.xlsx`), not YAML. The Excel format exists because **the human user edits config in Excel** — it must be easy to read, navigate, and modify with dropdowns, descriptions, and visual grouping. YAML (`dps_config_fallback.yaml`) is a legacy fallback only.

### Architecture: Two Scripts, One Workbook

| Component | File | Role |
|-----------|------|------|
| **Generator** | `generate_config_template.py` | Builds `dps_config.xlsx` from scratch with defaults, styling, dropdowns, and descriptions. Can also populate from an existing YAML via `--from-yaml`. |
| **Reader** | `scripts/shared_utils.py` → `load_config_xlsx()` | Parses `dps_config.xlsx` back into a Python dict identical to what the YAML would produce. Every pipeline script calls `load_config()` which auto-detects `.xlsx` vs `.yaml`. |
| **Workbook** | `dps_config.xlsx` | 19 sheets, one per config section. The human edits this file directly in Excel. |

### How the Excel is Structured

Every sheet follows one of three layout patterns. Understanding these is critical for modifying the generator or the reader.

**Pattern 1 — Settings (most sheets):** Three columns: `Setting | Value | Description`. The human changes cells in column B. Column C is read-only guidance. Sub-headers (rows starting with `# `) visually group related settings. Comment rows (`## ...`) provide inline help and are skipped by the parser.

**Pattern 2 — Lists:** Single-column lists of values (e.g., exclude patterns, section keywords, ignore lists, regex patterns). Items are in column A, one per row. The user adds/removes rows. Comment rows (`## ...`) are skipped.

**Pattern 3 — Maps / Multi-column data:** Key-value pairs or structured rows (e.g., custom style map, extraction patterns with type, metadata fields with key/label/source). Parsed by column position.

### Sheet Inventory (19 sheets)

| Sheet | Pattern | What the Human Configures |
|-------|---------|--------------------------|
| **README** | (instructions) | Not config — explains how to use the workbook |
| **Input** | Settings + List | Input directory, glob pattern, recursive flag, exclude patterns |
| **Output** | Settings | Output directory and sub-folder/filename for every step (0-9) |
| **Sections** | List (by category) | Keywords the profiler fuzzy-matches against H1 headings (purpose, scope, intent, controls, appendix) |
| **Headings** | List + Map + Settings | Built-in/custom heading styles, custom→standard style map, fake heading detection thresholds, heading level regex patterns |
| **Text Deletions** | Settings + List + Table | Enable/disable text deletion in Step 3, exact phrases to remove, **section deletions** (heading + body removal by heading name match) |
| **Cross References** | Settings + List | Detection toggles, profiler regex patterns, extraction phrase→type pairs, document name keywords |
| **Tables** | List (by type) | Keywords and min-column counts for classifying table types (control matrix, applicability, etc.) |
| **Classification** | Settings + List | Thresholds for Type A/B/D classification, Type C procedure keywords |
| **Profiling Flags** | Settings | Thresholds for ControlDense, HeadingVariance, TableDense flags |
| **Thresholds** | Settings | `max_characters` (chunk size limit), page estimation ratios, flagging thresholds |
| **Priority Scoring** | Settings + Map | Scoring weights per factor, per-document usage frequency overrides |
| **Search Terms** | Settings + List | Enable/mode/display for key term search, list of terms (each becomes a column in profiler output) |
| **Control Extraction** | Settings + List + Map | Bold requirement, patterns, whitelist/blacklist, guidance keywords, metadata triggers, heading detection |
| **Pipeline** | Table | Step number, name, script, enabled (TRUE/FALSE), description — one row per step |
| **Metadata** | Settings + List | Placement, styling, field definitions (key/label/source), URL lookup config, tag generation, static tags |
| **Docx2md** | Settings + List | Pure vs optimized mode, image handling, table strategy, heading cleanup, frontmatter fields |
| **Docx2jsonl** | Settings | Chunking params, acronym definitions file, tag mapping |
| **Acronym Finder** | Settings + List | Search params, detection pattern toggles, ignore list |

### Critical Conventions (Must Follow When Editing)

1. **Sub-headers (`# Title`) are structural delimiters.** The parser uses them to split rows into named blocks. Never delete, rename, or reorder sub-header rows — the reader matches on these names. Adding new sub-headers requires updating the corresponding `_parse_*_sheet()` function in `shared_utils.py`.

2. **Comment rows (`## ...`) are visual help only.** They are skipped by the parser. Safe to add, edit, or remove for clarity.

3. **Setting keys in column A must match exactly** what the pipeline code expects. The reader builds a dict using these keys. Renaming a key in the Excel without updating the reader and all consuming scripts will silently drop the setting.

4. **Boolean cells use Excel TRUE/FALSE** with dropdown validation. The reader coerces these. Do not use "Yes"/"No" or "1"/"0".

5. **Empty Value cells = use built-in default.** The pipeline code uses `.get(key, default)` everywhere. Leaving a cell blank is intentional and valid — it means "use the hardcoded default."

6. **List items: add/remove rows freely** within the correct sub-header block. The parser collects all non-empty, non-comment rows in that block.

7. **Generator and reader must stay in sync.** If you add a new setting to `generate_config_template.py`, you must also add parsing logic in `shared_utils.py`'s corresponding `_parse_*_sheet()` function, AND update the consuming script to read the new key. The three-way contract is: generator writes it → reader parses it → script uses it.

8. **Styling is for the human.** The generator applies blue headers, sub-header fills, description fonts, frozen panes, autofilter, auto-width, and data validation dropdowns. These make the Excel easy to use. The reader ignores all styling — it only reads cell values.

9. **New sheets require wiring in three places:** (a) a `_build_*` function in `generate_config_template.py`, (b) a call to it in `generate_config_workbook()`, (c) a `_parse_*_sheet()` function in `shared_utils.py` called from `load_config_xlsx()`.

### How to Add a New Config Setting

1. **Find the right sheet** in `generate_config_template.py` (the `_build_*` function).
2. **Add the setting** using `_add_setting(ws, "key_name", default_value, "Description for human")` inside the appropriate sub-header block. Add dropdown validation if boolean or enum.
3. **Update the reader** in `shared_utils.py` — find the matching `_parse_*_sheet()` function and ensure the key is extracted (most settings sheets use `_parse_settings_rows()` which auto-captures all Setting/Value pairs, but nested keys like `"tags.include_doc_type"` require explicit handling).
4. **Update the consuming script** to read the new key from the config dict.
5. **Regenerate** the Excel: `python generate_config_template.py` (or `--from-yaml dps_config_fallback.yaml` to preserve existing values).

### How to Add a New Config Sheet

1. Write a `_build_<name>_sheet(wb, cfg)` function in `generate_config_template.py` following existing patterns.
2. Call it from `generate_config_workbook()` in the desired tab order.
3. Write a `_parse_<name>_sheet(ws)` function in `shared_utils.py`.
4. Wire it into `load_config_xlsx()` with the sheet name string matching exactly what the generator creates.
5. Regenerate the Excel.

### Common Pitfalls

- **Editing the Excel while it's open in Excel** can create lock files (`~$dps_config.xlsx`). The pipeline's exclude patterns already filter these from input, but the config reader will fail if the lock file prevents access.
- **Merged cells break the parser.** The generator intentionally avoids merged cells. Never merge cells in the config workbook.
- **The generator overwrites the entire file.** Running `generate_config_template.py` replaces `dps_config.xlsx` completely. User customizations are lost unless the `--from-yaml` flag is used to seed values. There is no "merge" mode — the generator always writes a fresh workbook with defaults (or YAML-seeded values).
- **Dropdown validation is for guidance, not enforcement.** The reader does not validate values against allowed options. Invalid values may cause runtime errors in pipeline scripts, not at config load time.
